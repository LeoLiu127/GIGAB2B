# GIGAB2B 模板填表 UI 整合设计

## 目标

在不改变现有 Listing Creator & Optimizer 工作流、模板填表后端接口和 XLSM 处理能力的前提下，让两个前端模块形成一个视觉一致、入口互通、运行隔离的产品。

本次整合同时保留并复用现有三套主题：

- `light`：白色界面、黑色主操作；
- `saddle`：牛皮色界面、驼色主操作；
- `midnight`：暗色界面、蓝色主操作。

## 已确认的设计目标

- 主工作台继续使用现有三栏布局，业务内容和操作顺序不变。
- 主工作台只增加一个轻量的“Amazon 模板填表”入口。
- 模板填表继续通过 `/template-filler.html` 独立访问，不嵌入主工作台、不共享 React 业务状态。
- 模板填表页面采用已确认的三栏工作台视觉：左侧上传与处理、中间模板分析与策略、右侧变体和校验报告。
- 三个主题共用现有主题令牌和 `gigab2b-theme` 本地偏好；在任一页面切换主题，另一页面下次打开时保持一致。
- 后端能力、API 请求结构、响应结构、XLSM 下载流程、策略库和变体逻辑不因本次 UI 整合而改变。

## 方案比较

### 方案 A：独立页面，共享视觉基础（采用）

主工作台与模板填表保持两个入口。共享主题令牌、主题选择器的行为和少量顶栏样式；各自维护页面组件和业务状态。

优点：隔离最强、合并冲突少、模板模块可以持续独立演进、出现故障时不会阻塞主工作台。缺点：少量顶栏结构可能需要分别维护。

### 方案 B：模板页改写为 React 并共享 Header 组件（不采用）

可以减少顶栏重复，但会把当前稳定的静态 TypeScript 页面迁移到 React，扩大改动范围，并迫使两个入口共享更多构建和组件约束。本次仅为视觉整合，不值得承担迁移风险。

### 方案 C：模板功能嵌入主工作台标签页（不采用）

发现性最高，但会把上传、策略编辑、变体和报告状态带入原工作台，显著增加布局、路由、状态管理和回归风险，也违背“原 UI 只做微小改动”的约束。

## 架构与文件边界

### 共享基础

- `web/src/theme.ts`
  - 继续作为三套主题的唯一颜色令牌和本地存储键来源。
  - 增加可供 React 主应用和静态模板页共同调用的文档主题应用函数。
- `web/src/index.css`
  - 保留主工作台全局基础样式。
  - 只在确有必要时增加前缀明确的共享顶栏/模块切换样式，禁止模板页的表格和布局选择器进入该文件。
- `web/vite.config.ts`
  - 保持 `/index.html` 与 `/template-filler.html` 两个构建入口。

### 主工作台

- `web/src/App.tsx` 或 `web/src/components/Header.tsx`
  - 仅增加到 `/template-filler.html` 的轻量入口。
  - 不改变抓取、文案优化、图片生成、模板上传或下载业务状态。
  - 不把模板填表组件挂载到主应用 React 树。

### 模板填表页

- `web/template-filler.html`
  - 改为紧凑的三栏工作台语义结构。
  - 保留现有 DOM 功能节点的 `id`，避免改动后端调用与业务事件绑定。
- `web/src/template-filler.css`
  - 完整负责模板页布局和组件外观。
  - 所有颜色从 `--theme-*` 令牌读取；不得写入另一套独立主题色表。
  - 使用页面前缀或明确类名，避免污染主工作台。
- `web/src/template-filler.ts`
  - 保留现有 analyze、policy、fill、variant、report 和 download 行为。
  - 仅增加主题初始化、主题按钮绑定、现有系统状态展示和必要的 UI 状态切换。
- `web/src/template-filler-model.ts`
  - 继续承载纯格式化和兼容性逻辑；本次不改变后端数据模型。

### 后端

- `/api/template-filler/*`、`template_filler/*` 和 SQLite 策略库不因本次 UI 整合修改。
- 主工作台原接口不修改。
- 模板页可只读复用现有系统状态接口；若状态接口暂不可用，顶栏显示“状态不可用”，但不得阻止模板上传和填表。

## 页面结构

### 顶栏

- 左侧沿用 `Listing Creator & Optimizer for GIGAB2B` 品牌文字。
- 右侧保留三主题选择器和系统状态标签。
- 增加两个轻量模块入口：`文案/图片优化` 与 `Amazon 模板填表`；当前页面使用对应主题的主操作色。

### 左栏：模板与处理

- 站点：Amazon.co.uk（英国）。
- Seller Central 模板上传区。
- “展开同 Listing 全部变体”开关。
- “分析模板”“抓取并填表”操作。
- 当前处理进度与错误摘要。

### 中栏：分析与策略

- 模板识别结果：平台、站点、类目、工作表、SKU 行数。
- 字段 ID、策略类型、适用对象、默认值和模板允许值。
- 策略状态、结构漂移和保存操作。
- 表格采用紧凑行式布局，不使用营销式大卡片。

### 右栏：变体、校验与下载

- 显示预期子体数、实际抓取数、父子数量、变体主题和阻断原因。
- 当预期数量等于实际抓取数量时，额外不可访问关联编号只作为说明，不暗示缺少子体。
- 校验结果区分严格必填、运营必填、下拉不匹配、提醒和查询失败。
- 下载填写后的模板和 JSON 校验报告。

## 数据流与失败隔离

1. 模板页加载时读取 `gigab2b-theme` 并应用现有主题令牌。
2. 用户切换主题时只更新文档令牌和本地偏好，不调用后端。
3. 上传、分析、策略保存、填表和下载继续使用当前 `/api/template-filler/*` 契约。
4. 顶栏系统状态请求失败只影响状态标签，不影响模板工作流。
5. 模板页前端异常由模板页自身处理，不会影响 `/` 主应用加载。

## 后续模板功能改进的影响范围

### 低风险：默认不会影响原项目

- 新增 Amazon 类目策略、必填规则或下拉规则；
- 模板解析、字段映射、校验报告和变体主题推断改进；
- 模板页布局、筛选、报告展示和下载体验改进；
- 以上改动只触及 `template_filler/*`、`web/template-filler.html`、`web/src/template-filler-*` 和对应测试时，不会改变主工作台行为。

### 中风险：需要双系统回归

- 修改共享 `theme.ts`、`index.css`、`vite.config.ts` 或顶栏组件；
- 修改 `app.py` 的蓝图注册、下载目录或全局错误处理；
- 复用或调整主项目的 GIGA 变体采集函数；
- 这些改动可能影响两个入口，必须同时运行主工作台与模板页测试。

### 高风险：必须单独设计和迁移

- 把模板填表状态并入主 React 应用；
- 改写现有 GIGA API 认证、批量抓取或下载机制；
- 把本机 SQLite 策略库改为共享数据库；
- 修改输出文件目录、后台任务模型或多实例部署方式。

这些变化不属于本次 UI 整合，未来必须单独分支、单独规格和双系统回归。

- 主题在两个页面间保持同步。
- CABINET 与 CHAIR 均可完成上传、分析、策略读取、变体展开、填表、问题报告和下载。
